## 2. 잘한 점

### 기능별 폴더 구조 (응집도 높음!)

컴포넌트를 `(Auth)/`, `board/`, `mypage/`, `pokedex/`로 기능별로 묶은 것은 **높은 응집도(Cohesion)**의 좋은 예시예요. "관련된 코드는 가까이 두자"라는 원칙을 잘 지키고 있어요.

```
src/components/
├── (Auth)/          ← 인증 관련 코드만 모아놓음
│   ├── login.js
│   └── register.js
├── board/           ← 게시판 관련 코드만 모아놓음
│   ├── boardDetail.js
│   └── boardDetailUI.js
├── mypage/          ← 마이페이지 관련 코드만 모아놓음
│   ├── my.js
│   ├── myboard.js
│   └── myinfo.js
```

### UI와 로직의 분리

`boardDetail.js`(로직)와 `boardDetailUI.js`(화면 그리기)를 분리한 것은 **결합도를 낮추는** 훌륭한 시도예요! UI를 바꿔도 로직이 영향 받지 않고, 로직을 바꿔도 UI가 영향 받지 않아요. 이런 분리는 유지보수를 쉽게 만들어줘요.

```js
// boardDetail.js — 데이터를 준비하고
import { BoardDetailContent, CommentSection } from "./boardDetailUI.js";
contentArea.innerHTML = BoardDetailContent(dummydetailPost);

// boardDetailUI.js — 화면만 담당
export const BoardDetailContent = (post) => { ... };
export const CommentSection = (comments) => { ... };
```

### 나만의 파티 - 완성도 높은 CRUD 구현

`src/scripts/myparty.js`는 이 프로젝트에서 가장 완성도 높은 파일이에요! API 연동, 인증 처리, 에러 핸들링, 더미 데이터 폴백까지 모두 갖추고 있어요.

```js
// 토큰 없으면 더미 데이터로 폴백 — 개발 중에도 테스트 가능하도록 잘 처리!
if (!token) {
  console.info("[dev] 토큰 없음 → 더미 포켓몬 사용");
  pokemons = await fetchPokemonsByIds(DUMMY_IDS);
  return;
}
```

특히 `escapeHtml()` 함수를 직접 만들어 XSS 방지를 한 것은 보안 의식이 돋보여요:
```js
function escapeHtml(text) {
  const div = document.createElement("div");
  div.textContent = text;
  return div.innerHTML;
}
```

### 로그인/회원가입 폼 검증

```js
// 비밀번호 보기/숨기기 토글
eye.addEventListener('click', function () {
```

### 포켓몬 도감 - 검색 + 페이지네이션 + 반응형

`pokedex.js`와 `pokedexUI.js`에서 PokeAPI 연동, 한국어 이름 매핑, 검색, 페이지네이션, 사이드바 아코디언까지 복잡한 기능을 잘 구현했어요. `Promise.all`로 병렬 요청을 보내는 것도 성능을 고려한 좋은 코드예요.

```js
const details = await Promise.all(promises);
```

### 마이파티 반응형 CSS가 잘 구성됨

`pages/myparty.html`에서 `<style>` 태그 안에 CSS Grid + 미디어 쿼리를 활용해 데스크탑(2열) → 태블릿(1열, 3열 그리드) → 모바일(1열, 2열 그리드)로 잘 구성했어요.

```css
.myparty-pokemon-grid {

}
@media (max-width: 1024px) {

  }
}
@media (max-width: 768px) {
  .myparty-pokemon-grid {
    grid-template-columns: repeat(2, 1fr);  /* 모바일 */
  }
}
```

### SPA 네비게이션 pushState 패턴

`header.js`에서 `history.pushState`와 `PopStateEvent`를 사용한 SPA 네비게이션 패턴은 프레임워크 없이 직접 구현하기 어려운 건데, 잘 해냈어요!

```js
document.addEventListener('click', (e) => {
  if (e.target.classList.contains('login-btn')) {
    history.pushState(null, '', '/login');
    window.dispatchEvent(new PopStateEvent('popstate'));
  }
});
```

## 3. 개선하면 좋을 점

### 3-1. 게시글 상세 - DOM이 없을 때 initPostDetail 호출 (중요도: 높음)

**현재 상태:**

`src/main.js` 79~97번째 줄에서 `initPostDetail(2)`를 호출한 **뒤에** HTML을 `innerHTML`로 삽입하고 있어요.

```js
// src/main.js 86~97번째 줄
if (page.includes("detailPost.html")) {
  console.log(`2번 게시글 상세페이지 로드 중...`);
  initPostDetail(2);   // ← HTML이 아직 DOM에 없는 상태에서 호출!
}

document.getElementById('content').innerHTML = html;  // ← 여기서야 HTML이 들어감
```

`initPostDetail()` 안에서 `document.getElementById("postDetailContent")`로 DOM 요소를 찾는데, 이 시점에는 HTML이 아직 들어가기 전이라 `null`이 반환돼요.

**왜 개선하면 좋을까?**

이전 리뷰(3-7)에서 `board.js`의 `import()` 타이밍 문제를 지적했는데, 같은 패턴이 반복되고 있어요. HTML을 먼저 넣고, 그 다음에 초기화 함수를 호출해야 DOM 요소를 찾을 수 있어요.

**추천:**

```js
// HTML을 먼저 넣기
document.getElementById('content').innerHTML = html;

// 그 다음에 초기화
if (page.includes("detailPost.html")) {
  initPostDetail(postId);  // postId 변수 활용!
}
```

### 3-2. postId 하드코딩 — 항상 2번 게시글만 보임 (중요도: 높음)

**현재 상태:**

```js
// src/main.js 87~90번째 줄
if (page.includes("detailPost.html")) {
  //테스트용으로 2번 게시물을 불러옴 수정 꼭!! 해야함!!! postId로!!!
  console.log(`2번 게시글 상세페이지 로드 중...`);
  initPostDetail(2);  // ← 항상 2번만!
}
```

**왜 개선하면 좋을까?**

주석에도 "수정 꼭!! 해야함!!!"이라고 적혀 있죠! URL에서 `postId`를 이미 추출하고 있는데(52번째 줄 `const postId = pathParts[2]`) 사용하지 않고 있어요. 어떤 게시글을 눌러도 항상 같은 글이 보이면 사용자가 혼란스러워요.

**추천:**

```js
if (page.includes("detailPost.html")) {
  initPostDetail(postId);  // 이미 52번째 줄에서 추출한 postId 사용!
}
```

### 3-3. mypage의 my.js — 즉시 실행으로 인한 높은 결합도 (중요도: 높음)

**현재 상태:**

`src/components/mypage/my.js`는 import되는 순간 바로 DOM에 접근해요:

```js
// src/components/mypage/my.js 4~7번째 줄
const btnMyInfo = document.getElementById('btn-myinfo');
const btnMyBoard = document.getElementById('btn-myboard');
const btnLogout = document.getElementById('btn-logout');
const btnWithdraw = document.getElementById('btn-withdraw');

// 22번째 줄 — 바로 이벤트 등록
btnMyInfo.addEventListener('click', () => { ... });

// 50번째 줄 — 바로 실행
initMyInfo();
```

**왜 개선하면 좋을까?**

이 파일이 import되는 시점에 `btn-myinfo` 등의 DOM 요소가 반드시 존재해야 해요. 만약 HTML이 아직 로드되기 전에 이 파일이 실행되면 `Cannot read properties of null` 에러가 발생해요.

다른 파일들(`pokedex.js`, `myparty.js`)은 `initPokedex()`, `init()`처럼 **초기화 함수를 export**해서 적절한 시점에 호출하는 패턴을 쓰고 있어요. `my.js`만 이 패턴을 따르지 않아서 **결합도가 높아요** — HTML의 로드 타이밍에 강하게 의존하고 있거든요.

**추천:**

```js
// src/components/mypage/my.js — 초기화 함수로 감싸기
import { initMyInfo } from './myinfo.js';
import { initMyBoard } from './myboard.js';

export function initMyPage() {
  const btnMyInfo = document.getElementById('btn-myinfo');
  const btnMyBoard = document.getElementById('btn-myboard');
  const btnLogout = document.getElementById('btn-logout');
  const btnWithdraw = document.getElementById('btn-withdraw');

  // ... 이벤트 리스너 등록 ...

  initMyInfo();  // 기본 탭
}
```

그리고 `main.js`에서 적절한 시점에 호출:
```js
// src/main.js
if (current === "mypage") {
  document.getElementById('content').innerHTML = html;
  const { initMyPage } = await import('./components/mypage/my.js');
  initMyPage();
}
```

### 3-6. login.js에서 `==` 대신 `===` 사용 필요 (중요도: 중간)

**현재 상태:**

```js
// src/components/(Auth)/login.js 118~127번째 줄
if (login_id.value == '') {    // ← == 사용
  msg.style.display = 'block';
  msg.innerHTML = '아이디를 입력해주세요';
  ...
}

if (password.value == '') {    // ← == 사용
  msg.style.display = 'block';
  msg.innerHTML = '비밀번호를 입력해주세요';
  ...
}
```

**왜 개선하면 좋을까?**

`==`(느슨한 비교)는 **타입 변환**이 일어나서 예상치 못한 결과가 생길 수 있어요. 예를 들어 `0 == ''`는 `true`가 돼요! `===`(엄격한 비교)는 타입까지 같아야 `true`를 반환하므로 더 안전해요.

참고로 `register.js`에서는 `===`를 올바르게 사용하고 있어요. 동일한 프로젝트 내에서 일관성을 유지하는 것이 좋아요.

**추천:**

```js
if (login_id.value === '') {   // === 사용
if (password.value === '') {   // === 사용
```

### 3-8. boardDetailUI.js — 사용자 데이터를 innerHTML에 직접 삽입 (중요도: 중간)

**현재 상태:**

```js
// src/components/board/boardDetailUI.js 5~11번째 줄
export const Comment = (comment) => `
  <div ...>
    <span ...>${comment.authorNickname}</span>

  </div>

```

```js
// src/components/board/boardDetailUI.js 22~54번째 줄
export const BoardDetailContent = (post) => {
  return `
    <span ...>${post.category}</span>
    <h1 ...>${post.title}</h1>
    <span ...>${post.author}</span>
    <img src="${post.ImgUrls[0]}" ...>
```

**왜 개선하면 좋을까?**

**추천:**

```js
// src/utils/escapeHtml.js
export function escapeHtml(text) {
  const div = document.createElement("div");
  div.textContent = text;
  return div.innerHTML;
}
```

```js
// boardDetailUI.js
import { escapeHtml } from '../../utils/escapeHtml.js';

export const Comment = (comment) => `
  <span>${escapeHtml(comment.authorNickname)}</span>
  <p>${escapeHtml(comment.content)}</p>
`;
```

### 3-9. HTML 조각에 `<link>` 태그가 계속 반복됨 (중요도: 중간)

**현재 상태:**

이전 리뷰에서 `board.html`의 `<link>` 태그를 지적했는데, 새로 만든 페이지에서도 같은 패턴이 반복되고 있어요:

```html
<!-- pages/detailPost.html 1번째 줄 -->
<link rel="stylesheet" href="/src/style.css" />

<!-- pages/mypage.html 1번째 줄 -->
<link rel="stylesheet" href="/src/style.css" />
```

**추천:**

모든 `pages/*.html` 파일에서 첫 줄의 `<link>` 태그를 삭제하세요.

### 3-16. pokedexUI.js — `window` 전역 함수 사용 (중요도: 낮음)

**현재 상태:**

```js
// src/components/pokedex/pokedexUI.js 112번째 줄
window.toggleSidebar = function () { ... };

// src/components/pokedex/pokedex.js 101번째 줄
window.selectPokemon = async function (no) { ... };
```

```html
<!-- pages/pokedex.html 11번째 줄 -->
<div onclick="window.toggleSidebar()">
```

```js
// pokedexUI.js 25번째 줄
<div onclick="selectPokemon(${p.no})">
```

**왜 개선하면 좋을까?**

`window`에 함수를 붙이는 것은 **전역 변수 오염**이에요. 프로젝트가 커지면 다른 사람이 만든 코드의 함수 이름과 충돌할 수 있어요. 또한 HTML의 `onclick` 속성 대신 `addEventListener`를 사용하는 것이 현대적인 방식이에요.

지금은 프로젝트 규모가 작아서 문제 없지만, 나중을 위해 알아두면 좋아요.

**추천:**

이벤트 위임(Event Delegation) 패턴을 사용하면 `window` 전역 함수 없이도 가능해요:

```js
// pokedex.js — 사이드바 항목 클릭을 이벤트 위임으로 처리
const list = document.getElementById("sidebarList");
list.addEventListener("click", (e) => {
  const item = e.target.closest("[data-no]");
  if (item) selectPokemon(Number(item.dataset.no));
});
```

```js
// pokedexUI.js — onclick 대신 data 속성 사용
export const SidebarItem = (p) => `
  <div data-no="${p.no}" class="..." style="...">
    No.${p.no} ${p.name}
  </div>
`;
```

## 6. 마무리

이전 리뷰에서 빈 페이지였던 포켓덱스, 마이파티, 마이페이지가 **모두 구현 완료**된 것은 정말 대단한 발전이에요! 그리고 로그인/회원가입, 게시글 상세까지 — 프로젝트가 실제로 동작하는 서비스처럼 갖춰지고 있어요.

특히 인상적인 것들:
- **기능별 폴더 분류** — 응집도가 높은 구조를 직접 설계한 것
- **UI와 로직 분리** (boardDetail ↔ boardDetailUI) — 결합도를 낮추는 좋은 패턴
- **myparty.js의 CRUD** — API 연동, 인증, 에러 핸들링, XSS 방지까지 완성도 최고
- **반응형 디자인** — 모든 페이지에서 데스크탑/태블릿/모바일 대응

이번 리뷰의 핵심 키워드는 **"응집도는 높게, 결합도는 낮게"**예요:
- `myparty.js`, `pokedex.js`처럼 **초기화 함수를 export하는 패턴**이 결합도를 낮추는 좋은 방법이에요
- `my.js`처럼 import 시점에 바로 DOM에 접근하면 결합도가 높아져요
- `escapeHtml` 같은 공통 유틸리티를 분리하면 코드 재사용성이 높아져요

이 리뷰의 높음/중간 항목들은 대부분 간단한 수정이에요. 3-1(DOM 순서)과 3-2(postId)는 5분이면 끝나요! 규나인연 팀, 이 속도로 가면 프로젝트 완성이 눈앞이에요. 화이팅!
