## 2. 잘한 점

### UI와 로직의 분리 — 정말 큰 발전이에요!

이번에 가장 인상적인 변화는 **UI(화면 그리기)와 로직(데이터 처리)을 분리**한 것이에요! 이건 프로 개발자들도 중요하게 생각하는 설계 원칙인 **"관심사의 분리(Separation of Concerns)"**예요.

```
login.js    ← 로그인 로직 (API 호출, 유효성 검사)
loginUI.js  ← 로그인 화면 (HTML 생성)

register.js    ← 회원가입 로직
registerUI.js  ← 회원가입 화면
```

mypage 쪽도 마찬가지예요:
```
myinfo.js    ← 내 정보 로직
myinfoUI.js  ← 내 정보 화면

myboard.js    ← 내 게시글 로직
myboardUI.js  ← 게시글 카드 화면
```

이렇게 하면 "화면 디자인을 바꾸고 싶다"면 UI 파일만, "API를 바꾸고 싶다"면 로직 파일만 수정하면 돼요. 이게 바로 **응집도는 높이고, 결합도는 낮추는** 대표적인 방법이에요!

### API 중앙 관리 — 결합도를 확 낮췄어요!

이전 리뷰에서 "API URL이 10곳에 하드코딩"이라고 지적했는데, 이걸 `src/api/config.js`로 분리해서 해결했어요!

```js
// src/api/config.js
export const BASE_URL = import.meta.env.VITE_BASE_URL;
```

그리고 `src/api/user.js`에 모든 사용자 관련 API를 모아둔 것은 **"하나의 파일이 하나의 역할만 하도록"** 하는 좋은 설계예요:

```js
// src/api/user.js — 사용자 관련 API만 모아둠
export async function login(loginId, password) { ... }

export async function getMyPosts() { ... }
export async function getMyPocketmons() { ... }
```

### XSS 방지 — 보안 의식이 성장했어요!

`escapeHtml` 유틸리티를 만들어 사용자 입력 데이터를 안전하게 출력하고 있어요:

```js
// src/utils/escapeHtml.js
export function escapeHtml(text) {
  const div = document.createElement("div");
  div.textContent = String(text ?? "");
  return div.innerHTML;
}

// boardDetailUI.js에서 사용
${escapeHtml(categoryMap[post.category] ?? post.category)}
${escapeHtml(post.nickname)}
${escapeHtml(post.content.trim())}
```

`board.js`, `myboardUI.js`에서도 `escapeHtml`을 사용하고 있어요. 보안 의식이 많이 성장한 거예요!

## 3. 개선하면 좋을 점

### 3-1. write.js — `errorData` 참조 에러 (중요도: 높음 — 게시글 수정 실패 시 앱이 멈춰요!)

**현재 상태:**

```js
// src/components/board/write.js 236번째 줄
if (!editRes.ok) {
  throw new Error(errorData.message || '수정 실패');  // ← errorData가 선언되지 않았어요!
}
```

**왜 개선하면 좋을까?**

게시글 수정이 서버에서 실패하면(예: 네트워크 오류, 서버 에러) `errorData`를 찾으려고 하는데 이 변수는 코드 어디에도 선언되어 있지 않아요. 그래서 `ReferenceError: errorData is not defined`가 발생하면서 에러 처리 자체가 망가져요. 사용자는 "수정 실패"라는 안내 대신 빈 화면을 보게 돼요.

**추천:**

```js
if (!editRes.ok) {
  throw new Error('수정 실패');
}
```

또는 서버 응답에서 에러 메시지를 가져오고 싶다면:

```js
if (!editRes.ok) {
  const errorData = await editRes.json().catch(() => ({}));
  throw new Error(errorData.message || '수정 실패');
}
```

### 3-2. boardDetailUI.js — post.title과 comment.content에 escapeHtml 누락 (중요도: 높음 — XSS 보안 위험!)

**현재 상태:**

```js
// src/components/board/boardDetailUI.js 120번째 줄 (BoardDetailContent 함수)
<h1 class="text-3xl font-bold text-[#1a3a35] leading-tight flex-1">
  ${post.title}       <!-- ← escapeHtml 안 됨! -->
</h1>

// src/components/board/boardDetailUI.js 50번째 줄 (Comment 함수)
<p id="comment-content-${comment.commentId}" ...>
  ${comment.content}  <!-- ← escapeHtml 안 됨! -->
</p>
```

`post.content`에는 `escapeHtml`을 잘 적용했는데, `post.title`과 `comment.content`에는 빠졌어요.

**왜 개선하면 좋을까?**

누군가 게시글 제목에 `<script>alert('해킹!')</script>`을 넣으면, 그 게시글을 보는 모든 사용자의 브라우저에서 그 스크립트가 실행돼요. 이걸 **XSS(Cross-Site Scripting)**라고 하고, 웹 보안에서 가장 위험한 취약점 중 하나예요.

`post.content`에는 이미 `escapeHtml`을 적용한 것을 보면, 다른 곳에도 적용해야 한다는 걸 알고 있으면서 빠뜨린 거예요. 규칙은 간단해요: **사용자가 입력한 데이터는 무조건 escapeHtml로 감싸기!**

**추천:**

```js
// BoardDetailContent — title 수정
<h1 class="text-3xl font-bold text-[#1a3a35] leading-tight flex-1">
  ${escapeHtml(post.title)}
</h1>

// Comment — content 수정
<p id="comment-content-${comment.commentId}" ...>
  ${escapeHtml(comment.content)}
</p>
```

### 3-3. write.js — 모듈 최상위 `const token`이 stale(오래된) 값을 가짐 (중요도: 중간)

**현재 상태:**

```js
// src/components/board/write.js 17번째 줄
const token = localStorage.getItem('token');  // ← 모듈 로드 시 1번만 실행

// ...265번째 줄 (이미지 업로드)
const uploadRes = await fetch(`${BASE_URL}/images`, {
  method: 'POST',
  headers: { Authorization: `Bearer ${token}` },  // ← 오래된 토큰 사용!
  body: formData,
});
```

함수 내부(168번째 줄, 20번째 줄)에서는 `const token = localStorage.getItem('token')`으로 새로 가져오고 있는데, 이미지 업로드 부분(265줄)에서는 모듈 최상위의 오래된 `token`을 사용해요.

**왜 개선하면 좋을까?**

SPA에서 모듈은 한 번만 로드돼요. 만약 사용자가:
1. 로그아웃 → 재로그인 (새 토큰 발급)
2. 이미지 업로드 시도

이 경우 게시글 작성은 새 토큰으로 성공하지만, 이미지 업로드는 옛 토큰으로 시도해서 실패할 수 있어요. 이전 리뷰의 3-8 항목(pokedex.js의 isLoggedIn)과 동일한 패턴의 버그예요.

**추천:**

17번째 줄의 모듈 레벨 선언을 제거하세요:

```js
// 삭제: const token = localStorage.getItem('token');
```

265번째 줄에서 직접 가져오도록 변경하세요:

```js
document.getElementById('write-image').addEventListener('change', async (e) => {
  const file = e.target.files[0];
```

### 3-4. boardDetail.js — 하드코딩된 testPreset 데이터 (중요도: 중간)

**현재 상태:**

```js
// src/components/board/boardDetail.js 70~75번째 줄
const testPreset = {
  deckname: '테스트 파티',
  gender: 'woman',
  pocketmons: [1, 4, 7, 25, 39, 94],
};
const spriteMap = await fetchSprites(testPreset.pocketmons);

contentArea.innerHTML = BoardDetailContent(
  { ...post, preset: testPreset },   // ← 항상 테스트 데이터를 사용!
  currentUserId,

```

**왜 개선하면 좋을까?**

```js
// boardDetail.js — 6개의 window 전역 함수
window.toggleEditMode = (commentId) => { ... };

window.poketmonDelete = async function (event, id) { ... };
```

**왜 개선하면 좋을까?**

`header.js`에서 `window.logout`을 이벤트 위임으로 바꾼 것처럼, 다른 파일에서도 같은 패턴을 적용하면 좋아요. `window`에 함수를 붙이면:
1. **전역 공간 오염** — 다른 코드와 이름이 충돌할 수 있어요
2. **테스트 어려움** — window에 붙은 함수는 단위 테스트가 어려워요
3. **결합도 높음** — HTML의 onclick과 JS의 window 함수가 강하게 연결돼요

**추천:**

`header.js`에서 이미 구현한 **이벤트 위임 패턴**을 활용하세요. 예를 들어 boardDetail.js의 댓글 수정/삭제:

```js
// HTML에서 onclick 대신 data 속성 사용
`<button data-action="edit-comment" data-comment-id="${comment.commentId}">수정</button>`
`<button data-action="delete-comment" data-comment-id="${comment.commentId}">삭제</button>`

// JS에서 이벤트 위임
document.addEventListener('click', (e) => {
  const btn = e.target.closest('[data-action]');
  if (!btn) return;

  const action = btn.dataset.action;
  const commentId = btn.dataset.commentId;

  if (action === 'edit-comment') toggleEditMode(commentId);
  if (action === 'delete-comment') handleDeleteComment(commentId);
});
```

> **쉽게 말하면:** `onclick="함수이름()"`은 HTML과 JS가 서로의 이름을 알아야 해서 결합도가 높아요. `data-action`은 HTML은 "뭘 할지"만 적고, JS는 그걸 읽어서 처리하니까 더 느슨하게 연결돼요.

### 3-6. header.js — `location.reload()` 뒤의 코드가 실행 안 될 수 있어요 (중요도: 중간)

**현재 상태:**

```js
// src/components/header.js 83~91번째 줄
document.addEventListener('click', (e) => {
  if (e.target.classList.contains('logout-btn')) {
    localStorage.removeItem('token');
    localStorage.removeItem('userId');
    location.reload();               // ← 페이지를 새로고침!

    window.location.href = '/';      // ← 이 줄은 실행될 수도, 안 될 수도!
  }
});
```

**왜 개선하면 좋을까?**

`location.reload()`는 페이지를 새로고침하라는 명령이에요. 브라우저가 이 명령을 처리하기 시작하면 그 아래 코드(`window.location.href = '/'`)는 실행이 보장되지 않아요. 의도가 "로그아웃 후 홈으로 이동"이라면 `reload` 대신 `location.href`만 사용하면 돼요.

**추천:**

```js
document.addEventListener('click', (e) => {
  if (e.target.classList.contains('logout-btn')) {
    localStorage.removeItem('token');
    localStorage.removeItem('userId');
    window.location.href = '/';    // ← 홈으로 이동 (새로고침 효과 포함!)
  }
});
```

`window.location.href = '/'`는 페이지를 홈으로 이동시키면서 자동으로 새로고침이 되기 때문에, `location.reload()`가 필요 없어요.

### 3-7. main.js — import 경로 대소문자 불일치 (중요도: 중간 — 배포 환경에서 오류!)

```js
import { Register, initRegister } from './components/Auth/register.js';
```

> **쉽게 말하면:** "포켓몬"이라고 적었는데 상대방은 "포켓몬"이라고 읽는 건 같은 말이지만, 기계(Linux)는 정확히 같은 글자만 인정해요!

### 3-11. board.js — dummyData가 100줄 남아있어요 (중요도: 낮음)

**현재 상태:**

```js
// src/components/board/board.js 1~105번째 줄
const dummyData = [
  { postId: 1, title: '카비곤 육성 팁 공유합니다!', ... },
  { postId: 2, title: '레전드 포켓몬 추천해주세요', ... },
  // ... 10개 항목, 약 100줄
];
```

API 연동이 완료되었지만, API 실패 시 fallback으로 dummyData를 반환하고 있어요:

```js
const getPosts = async () => {
  try {
    // ... API 호출 ...
    return Array.isArray(posts) ? posts : dummyData;  // ← API 실패 시 더미 데이터
  } catch (error) {
    return dummyData;
  }
};
```

**왜 개선하면 좋을까?**

개발 중에는 편리하지만, 실제 서비스에서 API가 실패했을 때 "가짜 게시글"이 보이면 사용자가 혼란스러워해요. 진짜 게시글인 줄 알고 클릭하면 없는 게시글이라 에러가 나요.

**추천:**

```js
const getPosts = async () => {
  try {
    const response = await fetch(`${BASE_URL}/posts`, { method: 'GET' });
    if (!response.ok) throw new Error('Failed to fetch posts');
    const data = await response.json();
    const posts = data.data?.content ?? data.data ?? [];
    return Array.isArray(posts) ? posts : [];
  } catch (error) {
    console.error('Error fetching posts:', error);
    return [];  // ← 빈 배열 반환 + "게시글을 불러오지 못했습니다" 메시지 표시
  }
};
```
