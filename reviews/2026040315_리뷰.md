## 2. 잘한 점

### 컴포넌트 분리 구조

`header.js`, `footer.js`, `board.js`처럼 기능별로 파일을 나눈 것은 정말 좋은 습관이에요! Vanilla JS에서 이렇게 컴포넌트를 분리하는 건 쉽지 않은데, `export function` 문법을 활용해서 깔끔하게 구현했어요.

```js
// src/components/header.js
export function Header() {
  return `<header class="header">...</header>`;
}
```

```js
// src/main.js
import { Header } from './components/header.js';
import { Footer } from './components/footer.js';
```

ES 모듈의 `import/export`를 제대로 활용하고 있어요. 훌륭합니다!

### SPA 방식의 페이지 로딩

`main.js`에서 `fetch()`를 사용해 HTML 파일을 동적으로 불러오는 방식은 도전적이고 훌륭한 시도예요! URL 경로에 따라 다른 페이지를 로드하는 라우팅 로직도 잘 작성되어 있어요.

```js
async function loadPage() {
  const path = window.location.pathname;
  let page = './pages/pokedex.html';
  if (path.includes('board')) page = './pages/board.html';
  // ...
  const res = await fetch(page);
  const html = await res.text();
  document.getElementById('content').innerHTML = html;
}
```

초보자가 `async/await`와 `fetch()`를 이렇게 활용하는 것은 쉽지 않은 건데, 잘 해냈어요!

### 반응형 디자인 구현

`layout.css`에서 태블릿(1024px)과 모바일(768px)에 대한 미디어 쿼리를 적용했고, 데스크탑에서는 네비게이션 바를, 모바일에서는 햄버거 메뉴와 사이드바를 보여주는 구조를 잘 설계했어요.

```css
@media (max-width: 1024px) {
  .nav { display: none; }
  .login-btn { display: none; }
  .icon-btn, .menu-btn { display: block; }
}
```

반응형을 처음부터 고려한 것은 프로 개발자의 마인드예요!

### 게시판 기능의 완성도

`board.js`에서 카테고리 필터링, 검색, 페이지네이션까지 구현한 것은 대단해요! 더미 데이터를 사용해서 UI를 먼저 만들어보는 것도 좋은 개발 방식이에요.

```js
// 카테고리별 색상 매핑
const categoryColors = {
  자유게시판: 'text-[#00bba7] bg-[#e6f7f5]',
  질문게시판: 'text-pink-500 bg-pink-50',
  '파티 공유': 'text-amber-500 bg-amber-50',
  공략: 'text-blue-500 bg-blue-50',
};
```

카테고리별로 다른 색상을 적용한 것도 UX 관점에서 아주 좋아요!

### 사이드바 열기/닫기 + 오버레이

사이드바 열기, 닫기 버튼, 배경 오버레이 클릭 시 닫기까지 3가지 인터랙션을 모두 구현했어요. 사용자 경험을 세심하게 고려한 것이 느껴져요.

```js
// 열기
menuBtn.addEventListener('click', () => { sidebar.classList.add('active'); });
// 닫기 버튼
closeBtn?.addEventListener('click', () => { sidebar.classList.remove('active'); });
// 배경 클릭
overlay.addEventListener('click', () => { sidebar.classList.remove('active'); });
```

## 3. 개선하면 좋을 점

### 3-1. `<postlist>` 커스텀 태그 대신 표준 HTML 태그 사용 (중요도: 높음)

**현재 상태:**

`pages/board.html` 148번째 줄에서 `<postlist>`라는 존재하지 않는 HTML 태그를 사용하고 있어요.

```html
<!-- pages/board.html 148번째 줄 -->
<postlist></postlist>
```

```js
// src/components/board.js 118번째 줄
const postlist = document.querySelector('postlist');
postlist.style = 'display:flex; flex-direction:column; gap:16px;';
```

**왜 개선하면 좋을까?**

`<postlist>`는 HTML에 존재하지 않는 태그예요. 브라우저는 모르는 태그를 만나면 일단 `HTMLUnknownElement`로 처리하기 때문에 동작은 하지만, 이건 HTML 표준을 따르지 않는 방식이에요. 스크린 리더(시각장애인을 위한 도구)나 검색 엔진이 이 요소의 역할을 이해할 수 없어요.

**추천:**

```html
<!-- pages/board.html -->
<div id="postlist"></div>
```

```js
// src/components/board.js
const postlist = document.getElementById('postlist');
```

표준 `<div>` 태그에 `id`를 붙여서 사용하면 명확하고 안전해요.

### 3-2. `menuBtn` 이벤트 리스너 중복 등록 (중요도: 높음)

**현재 상태:**

`src/main.js`에서 같은 `menuBtn`에 클릭 이벤트를 **두 번** 등록하고 있어요.

```js
// src/main.js 16~25번째 줄 (첫 번째 등록)
const menuBtn = document.getElementById('menuBtn');
if (menuBtn) {
  menuBtn.addEventListener('click', () => {
    sidebar.classList.add('active');
    overlay.classList.add('active');
  });
}

// src/main.js 87~112번째 줄 (initSidebar 함수에서 두 번째 등록)
function initSidebar() {
  const menuBtn = document.getElementById('menuBtn');
  // ...
  menuBtn.addEventListener('click', () => {
    sidebar.classList.add('active');
    overlay.classList.add('active');
  });
}
initSidebar();
```

**왜 개선하면 좋을까?**

`addEventListener`은 호출할 때마다 이벤트 리스너를 **추가**해요. 같은 동작을 하는 리스너가 2개 달리면, 버튼 한 번 클릭했을 때 같은 코드가 2번 실행돼요. 지금은 `add('active')`를 두 번 하는 거라 눈에 보이는 차이가 없지만, 나중에 로직을 변경하면 예상치 못한 버그가 생길 수 있어요.

**추천:**

16~25번째 줄의 첫 번째 이벤트 등록 코드를 삭제하고, `initSidebar()` 함수에서만 관리하세요. `initSidebar()`가 더 완전한 코드(열기, 닫기, 오버레이 클릭)를 포함하고 있으니까요.

```js
// src/main.js — 16~25번째 줄 삭제
// 아래 코드를 모두 지워주세요:
// const menuBtn = document.getElementById('menuBtn');
// const sidebar = document.getElementById('sidebar');
// const overlay = document.getElementById('overlay');
// if (menuBtn) { ... }
```

### 3-3. `innerHTML`에 사용자 데이터 삽입 시 XSS 위험 (중요도: 높음)

**현재 상태:**

`src/components/board.js` 138번째 줄에서 게시글 데이터를 `innerHTML`로 삽입하고 있어요.

```js
// src/components/board.js 138번째 줄
postElement.innerHTML = `
  <span ...>${post.category}</span>
  <p ...>${post.title}</p>
  <span ...>${post.user}</span>
  // ...
`;
```

**왜 개선하면 좋을까?**

지금은 `dummyData`(개발용 가짜 데이터)를 사용하고 있어서 안전하지만, 나중에 API에서 실제 사용자가 작성한 데이터를 가져오면 위험해져요. 만약 누군가 게시글 제목에 `<script>alert('해킹!')</script>`라고 적으면, 그 코드가 그대로 실행될 수 있어요. 이걸 **XSS(Cross-Site Scripting, 크로스 사이트 스크립팅)** 공격이라고 해요.

쉽게 말하면, `innerHTML`은 HTML 코드를 그대로 해석하기 때문에 악성 코드도 실행할 수 있어요. `textContent`는 순수 텍스트로만 처리하기 때문에 안전해요.

**추천:**

사용자 데이터가 들어가는 부분은 `textContent`로 넣어주세요:

```js
// 안전한 방식
const titleEl = postElement.querySelector('.post-title');
titleEl.textContent = post.title;  // HTML 코드가 들어와도 그냥 텍스트로 표시
```

또는, HTML 구조는 `innerHTML`로 만들되, 사용자 데이터는 이후에 `textContent`로 넣는 방식도 좋아요. API를 연동할 때 반드시 이 부분을 고쳐주세요!

### 3-4. `<main>` 태그 중첩 (중요도: 중간)

**현재 상태:**

`src/main.js`에서 `<main class="main">`을 만들고, 그 안에 `pages/board.html`의 내용을 넣는데, `board.html`에도 `<main>` 태그가 있어요.

```js
// src/main.js 8~14번째 줄
app.innerHTML = `
  ${Header()}
  <main class="main">
    <div id="content" class="content"></div>
  </main>
  ${Footer()}
`;
```

```html
<!-- pages/board.html 3번째 줄 -->
<main>
  <!-- 게시판 내용 -->
</main>
```

결과적으로 `<main>` 안에 `<main>`이 들어가게 돼요.

**왜 개선하면 좋을까?**

HTML 표준에서 `<main>` 태그는 한 페이지에 **하나만** 있어야 해요. `<main>`은 "이 페이지의 주요 콘텐츠가 여기에요"라고 알려주는 시맨틱(의미를 가진) 태그인데, 두 개가 있으면 브라우저가 혼란스러워해요.

**추천:**

`pages/board.html`에서 `<main>` 태그를 제거하고, 내용만 남겨주세요:

```html
<!-- pages/board.html — <main> 태그 제거 -->
<link rel="stylesheet" href="/src/style.css" />
<div class="flex flex-col text-left">
  <h1 class="text-3xl font-bold text-[#1a3a35]">게시판</h1>
  <!-- ... -->
</div>
```

`<main>` 태그는 `main.js`에서 이미 만들어주고 있으니까요.

### 3-5. `board.html` HTML 조각 안에 `<link>` 태그 포함 (중요도: 중간)

**현재 상태:**

```html
<!-- pages/board.html 1번째 줄 -->
<link rel="stylesheet" href="/src/style.css" />
```

**왜 개선하면 좋을까?**

`board.html`은 독립된 HTML 문서가 아니라, `main.js`에서 `fetch()`로 가져와서 `innerHTML`에 삽입되는 **HTML 조각(fragment)**이에요. `<link>` 태그는 보통 `<head>` 안에 들어가야 하는데, 이렇게 `<div>` 안에 삽입되면 비표준적인 방식이에요.

게다가 `style.css`는 이미 `main.js`에서 Tailwind CSS를 불러오는 역할을 하고 있어서, `board.html`에서 다시 불러올 필요가 없어요.

**추천:**

`pages/board.html` 1번째 줄의 `<link>` 태그를 삭제해주세요. Tailwind CSS는 이미 `index.html` → `main.js` → `style.css` 경로로 로드되고 있어요.

### 3-6. 인라인 `style` 속성과 Tailwind 클래스 혼용 (중요도: 중간)

**현재 상태:**

`pages/board.html`에서 같은 요소에 인라인 `style`과 Tailwind 클래스를 동시에 사용하고 있어요.

```html
<!-- pages/board.html 14~16번째 줄 -->
<div
  class="shrink-0 bg-white rounded-xl max-[1025px]:w-full! ..."
  style="display: flex; flex-direction: column; width: 320px; height: 370px; padding: 24px 24px 0 24px; gap: 16px"
>
```

```html
<!-- pages/board.html 136~137번째 줄 -->
<postElement
  className="bg-white rounded-2xl border ..."
  style="display:flex; min-height:181px; padding:24px 24px 16px 24px; ..."
>
```

**왜 개선하면 좋을까?**

두 가지 방식으로 스타일을 적용하면 관리가 어려워져요. "이 요소의 padding을 바꾸려면 class를 고쳐야 하나, style을 고쳐야 하나?" 헷갈리게 돼요. 또한 인라인 `style`은 우선순위가 가장 높아서, Tailwind 클래스로 반응형 스타일을 적용해도 `style` 속성이 이겨버릴 수 있어요.

**추천:**

가능하면 Tailwind 클래스 하나로 통일해주세요:

```html
<!-- 수정 전 -->
<div class="shrink-0 bg-white rounded-xl"
     style="display: flex; flex-direction: column; width: 320px; padding: 24px;">

<!-- 수정 후 — Tailwind로 통일 -->
<div class="shrink-0 bg-white rounded-xl flex flex-col w-80 p-6">
```

Tailwind에서 제공하지 않는 아주 특수한 값만 `style`로 넣으면 돼요.

### 3-7. 동적 `import()` 타이밍 문제 (중요도: 중간)

**현재 상태:**

```js
// src/main.js 49~53번째 줄
if (current === 'board') {
  import('./components/board.js');  // await 없이 호출
}

document.getElementById('content').innerHTML = html;
```

**왜 개선하면 좋을까?**

`import()`는 비동기(async) 함수여서 Promise를 반환해요. 여기서는 `await` 없이 호출하고 있어서, `board.js`의 코드가 언제 실행될지 정확히 보장되지 않아요. `board.js` 안에서 `document.querySelector('postlist')`로 DOM 요소에 접근하는데, 만약 HTML이 아직 `innerHTML`에 들어가기 전에 `board.js`가 실행되면 에러가 발생해요.

현재는 우연히 타이밍이 맞아서 동작하고 있지만, 브라우저나 네트워크 상황에 따라 문제가 생길 수 있는 불안정한 코드예요.

**추천:**

HTML을 먼저 넣고, 그 다음에 `import`를 실행하세요:

```js
// src/main.js — 수정 후
document.getElementById('content').innerHTML = html;

if (current === 'board') {
  await import('./components/board.js');
}
```

이렇게 하면 HTML이 확실히 DOM에 들어간 후에 `board.js`가 실행돼요.

### 3-8. CSS 리셋 스타일 중복 (중요도: 낮음)

**현재 상태:**

```css
/* src/styles/layout.css 3~7번째 줄 */
* {
  margin: 0;
  padding: 0;
  box-sizing: border-box;
}
```

**왜 개선하면 좋을까?**

Tailwind CSS v4는 자체적으로 **Preflight**라는 리셋 스타일을 포함하고 있어요. `@import 'tailwindcss'`를 하면 자동으로 `margin: 0`, `padding: 0`, `box-sizing: border-box` 등이 적용돼요. `layout.css`에서 같은 리셋을 또 해주고 있으니 중복이에요.

**추천:**

`layout.css`의 `* { ... }` 리셋 부분을 삭제해도 돼요. Tailwind CSS가 알아서 해주니까요.

### 3-9. 검색 필터 `filter-type` 미사용 (중요도: 낮음)

**현재 상태:**

`board.html`에 검색 유형을 선택하는 `<select>` 요소가 있어요:

```html
<!-- pages/board.html 119~126번째 줄 -->
<select id="filter-type" class="...">
  <option value="title">제목</option>
  <option value="author">작성자</option>
</select>
```

하지만 `board.js`의 검색 로직에서는 이 값을 전혀 사용하지 않아요:

```js
// src/components/board.js 239~244번째 줄
const keyword = document.getElementById('search-input').value.trim().toLowerCase();
const filtered = dummyData.filter((post) =>
  post.title.toLowerCase().includes(keyword) ||
  post.user.toLowerCase().includes(keyword)
);
```

**왜 개선하면 좋을까?**

사용자가 "작성자"를 선택하고 검색해도 제목+작성자 모두에서 검색돼요. UI에 선택 옵션이 있는데 실제로 동작하지 않으면 사용자가 혼란을 느낄 수 있어요.

**추천:**

```js
const filterType = document.getElementById('filter-type').value;
const filtered = dummyData.filter((post) => {
  if (filterType === 'title') return post.title.toLowerCase().includes(keyword);
  if (filterType === 'author') return post.user.toLowerCase().includes(keyword);
  return false;
});
```

### 3-10. 사이드바 `.active` 스타일 CSS 중복 (중요도: 낮음)

**현재 상태:**

`layout.css`에서 사이드바 링크의 `.active` 스타일이 두 군데에 작성되어 있어요:

```css
/* src/styles/layout.css 182~192번째 줄 */
.sidebar-nav a {
  .active {        /* ← 중첩(nesting) 방식 */
    background: #cde5ec;
    border-radius: 12px;
    font-weight: bold;
  }
}

/* src/styles/layout.css 202~206번째 줄 */
.sidebar-nav a.active {  /* ← 일반 방식 */
  background: #cde5ec;
  border-radius: 12px;
  font-weight: bold;
}
```

**왜 개선하면 좋을까?**

같은 스타일이 두 번 적혀 있어요. 또한 182번째 줄의 중첩(nesting) 방식은 `.sidebar-nav a` 안에 `.active`라는 **자식 요소**를 찾는 것이라서, `a.active`와 의미가 달라요. 실제로 원하는 동작은 202번째 줄의 `a.active`예요.

**추천:**

182~192번째 줄의 중첩된 `.active` 부분을 삭제하고, 202번째 줄의 `a.active`만 남기세요.
