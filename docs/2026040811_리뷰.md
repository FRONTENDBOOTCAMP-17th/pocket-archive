## 2. 잘한 점

### 이전 리뷰 피드백을 대부분 반영했어요!

이전 리뷰에서 지적한 **13개 항목 중 12개**를 수정했어요. 특히 "높음" 등급이었던 `const` 재할당, `JSON.stringify` 누락, `setTimeout` 문제, 비밀번호 `console.log` 같은 **치명적인 버그들을 모두 해결**한 것은 정말 대단해요! 리뷰를 꼼꼼히 읽고 하나하나 고친 것이 느껴져요.

### 게시판 CRUD를 완성했어요!

게시글 목록 조회, 상세 보기, 작성, 수정, 삭제, 좋아요, 댓글 CRUD까지 — 게시판의 핵심 기능을 모두 구현했어요. 특히 `write.js`에서 카테고리 선택, 파티 프리셋 연동까지 넣은 것은 기획 의도를 잘 살린 거예요.

```js
// write.js — 게시글 작성 후 자동 발행까지!
const postId = data.data?.postId;
if (postId) {
  const publishRes = await fetch(`${BASE_URL}/posts/${postId}/publish`, {
    method: 'PUT',
    headers: { Authorization: `Bearer ${token}` },
  });
}
```

### XSS 방지 유틸리티를 만들고 적용했어요!

`escapeHtml.js` 유틸리티를 만들어서 `board.js`, `boardDetailUI.js`, `myboard.js`에 적용한 것은 **보안 의식이 생겼다는 증거**예요! 이전 리뷰에서 XSS를 지적했을 때 "그냥 그 부분만 고치자"가 아니라 "재사용 가능한 유틸리티를 만들자"라고 판단한 것은 개발자답게 사고한 거예요.

```js
// src/utils/escapeHtml.js — 깔끔하고 안전한 구현!
export function escapeHtml(text) {
  const div = document.createElement("div");
  div.textContent = String(text ?? "");
  return div.innerHTML;
}
```

### 동적 import로 코드 스플리팅을 구현했어요!

`main.js`에서 페이지별로 필요한 모듈만 동적으로 불러오는 것은 **성능 최적화**의 좋은 예시예요. 처음 페이지를 열 때 모든 코드를 다운로드하지 않고, 해당 페이지에 갈 때만 불러와요.

```js
// main.js — 게시판 페이지에 들어갈 때만 board.js를 불러옴
if (current === 'board' && page.includes('board.html')) {
  const { initBoard } = await import('./components/board/board.js');
  initBoard();
}
```

### PokeAPI 캐싱으로 성능을 개선했어요!

`pokedex.js`에서 `Map`을 사용해 한 번 불러온 포켓몬 데이터를 캐싱하는 것은 **불필요한 네트워크 요청을 줄이는** 훌륭한 패턴이에요.

```js
// pokedex.js — 이미 불러온 데이터는 다시 안 불러와요!
const pokemonCache = new Map();

async function fetchPokemonDetail(no) {
  if (pokemonCache.has(no)) {
    return pokemonCache.get(no);  // ← 캐시에 있으면 바로 반환!
  }
  // ... API 호출 후 캐시에 저장
  pokemonCache.set(no, data);
  return data;
}
```

### Promise.all과 Promise.allSettled 사용

여러 API를 동시에 호출할 때 `Promise.all` (myinfo.js)과 `Promise.allSettled` (pokedex.js)를 적절히 사용했어요. "모두 성공해야 할 때"와 "일부 실패해도 괜찮을 때"를 구분한 것은 비동기 프로그래밍에 대한 이해가 깊어졌다는 뜻이에요!

```js
// myinfo.js — 세 API를 동시에 호출 (하나라도 실패하면 에러)
const [userRes, postsRes, catchRes] = await Promise.all([...]);

// pokedex.js — 일부 포켓몬 데이터 실패해도 나머지는 표시
const details = await Promise.allSettled(promises);
details.forEach((result) => {
  if (result.status === 'fulfilled' && result.value) { ... }
});
```

### 마이파티 기능이 완성도 있게 구현되었어요!

트레이너 카드 미리보기, 포켓몬 슬롯 선택, 파티 프리셋 저장/수정/삭제, 검색까지 — `myparty.js`의 기능 완성도가 매우 높아요. 특히 같은 포켓몬을 다른 슬롯에 넣을 때 **스왑(교환)하는 로직**은 게임 UX를 잘 이해하고 구현한 거예요.

```js
// myparty.js — 이미 배치된 포켓몬을 다른 슬롯에 놓으면 자리를 바꿔요
if (duplicatedIndex !== -1) {
  if (duplicatedIndex === selectedSlot) {
    party[selectedSlot] = null;       // 같은 슬롯 클릭 → 제거
  } else {
    const temp = party[selectedSlot]; // 다른 슬롯 → 교환
    party[selectedSlot] = party[duplicatedIndex];
    party[duplicatedIndex] = temp;
  }
}
```

## 3. 개선하면 좋을 점

### 3-1. register.js — 중복확인 없이 회원가입이 가능한 버그 (중요도: 높음 — 가입 검증 우회!)

**현재 상태:**

```js
// src/components/Auth/register.js — initRegister() 안
let validNickname = true;   // ← 초기값이 true
let validId = true;          // ← 초기값이 true

// 중복확인 후: validNickname = !result.data.exists;
// (닉네임이 사용 가능하면 true, 이미 있으면 false)

// 입력값 변경 시:
nickname.addEventListener('input', () => {
  validNickname = true;      // ← true로 리셋!
  updateButtonState();
});

// 버튼 활성화 조건:
function updateButtonState() {
  if (validNickname === true && validId === true) {
    signupBtn.disabled = false;   // ← 활성화!
  }
}
```

**왜 개선하면 좋을까?**

시나리오를 따라가 보면 문제가 보여요:

1. 닉네임 입력칸에 아무거나 타이핑 → `validNickname = true` (리셋인데 true)
2. 아이디 입력칸에 아무거나 타이핑 → `validId = true` (리셋인데 true)
3. `updateButtonState()` 호출 → 둘 다 `true` → **버튼 활성화!**
4. 중복확인을 한 번도 하지 않았는데 **회원가입이 가능해요!**

문제의 원인은 `true`가 두 가지 뜻을 동시에 갖고 있기 때문이에요:
- "중복확인 완료 + 사용 가능" = `true`
- "아직 확인 안 함 (초기값/리셋)" = `true`

이 두 상태를 구분하지 못해서 버그가 생긴 거예요.

**추천:**

```js
// "아직 확인 안 함" = false, "확인 완료 + 사용 가능" = true
let validNickname = false;   // ← false로 시작!
let validId = false;          // ← false로 시작!

// 입력값 변경 시 — 재확인 필요하므로 false
nickname.addEventListener('input', () => {
  validNickname = false;      // ← false로 리셋!
  updateButtonState();
});

id.addEventListener('input', () => {
  validId = false;            // ← false로 리셋!
  updateButtonState();
});

// 버튼 활성화 조건 — 둘 다 확인 완료일 때만
function updateButtonState() {
  const signupBtn = document.getElementById('signupBtn');
  signupBtn.disabled = !(validNickname && validId);
}
```

> **쉽게 말하면:** 콘서트 입장할 때 "티켓 확인 했음" 도장이 필요해요. 처음에는 도장이 없는 상태(`false`)에서 시작해야지, 도장이 찍힌 상태(`true`)에서 시작하면 누구나 통과할 수 있어요!

### 3-2. boardDetailUI.js — 댓글 내용(comment.content)과 게시글 제목(post.title)에 escapeHtml 미적용 (중요도: 높음 — XSS 보안!)

**현재 상태:**

```js
// src/components/board/boardDetailUI.js 37번째 줄 — Comment 함수
<p id="comment-content-${comment.commentId}" class="...">${comment.content}</p>
//                                                        ^^^^^^^^^^^^^^^^
//                                         사용자가 작성한 댓글 내용이 그대로 들어가요!

// 69번째 줄 — BoardDetailContent 함수
<h1 class="text-3xl font-bold text-[#1a3a35] leading-tight">
  ${post.title}    // ← 게시글 제목도 이스케이프 없이!
</h1>
```

같은 파일에서 `category`, `author`, `content(본문)`에는 `escapeHtml`을 잘 적용했는데, **`comment.content`**와 **`post.title`**에는 빠뜨렸어요.

**왜 개선하면 좋을까?**

누군가 댓글에 이런 내용을 작성하면:
```
<img src=x onerror="alert('해킹!')">
```

### 3-3. boardDetail.js — 댓글 수정 시 onclick에 originalContent 직접 삽입 (중요도: 높음 — XSS + 동작 오류!)

**현재 상태:**

```js
// src/components/board/boardDetail.js 122~140번째 줄
window.toggleEditMode = (commentId) => {
  const contentP = document.getElementById(`comment-content-${commentId}`);
  const originalContent = contentP.innerText;

  // 2. 버튼을 '저장/취소'로 교체
  btnGroup.innerHTML = `
    <button onclick="saveEditComment(${commentId}, '${originalContent}')" ...>저장</button>
    <button onclick="cancelEditMode(${commentId}, '${originalContent}')" ...>취소</button>
  `;
};
```

**왜 개선하면 좋을까?**

`originalContent`가 HTML `onclick` 속성 안에 직접 들어가고 있어요. 만약 댓글 내용에 작은따옴표(`'`)가 포함되어 있으면:

```
댓글: It's a good day!
→ onclick="saveEditComment(5, 'It's a good day!')"
                                   ^ 여기서 문자열이 끊겨요!
```

작은따옴표가 `onclick`의 문자열을 깨뜨려서 **JavaScript 에러**가 발생하고, 더 심하면 **XSS 공격**에 악용될 수 있어요.

**추천:**

`onclick` HTML 속성 대신 `addEventListener`를 사용하면 이 문제가 완전히 사라져요:

```js
window.toggleEditMode = (commentId) => {
  const contentP = document.getElementById(`comment-content-${commentId}`);
  const btnGroup = document.getElementById(`comment-btns-${commentId}`);
  const originalContent = contentP.innerText;

  // textarea 교체 (동일)
  contentP.innerHTML = `
    <textarea id="edit-input-${commentId}" 
      class="w-full p-3 mt-2 border border-gray-200 rounded-lg focus:outline-none focus:border-[#05B29F] text-[14px] resize-none"
      rows="3">${escapeHtml(originalContent)}</textarea>
  `;

  // 버튼 — onclick 대신 addEventListener 사용!
  btnGroup.innerHTML = `
    <button id="save-btn-${commentId}" class="text-[11px] text-[#05B29F] font-bold">저장</button>
    <button id="cancel-btn-${commentId}" class="text-[11px] text-gray-400 font-bold">취소</button>
  `;

  document.getElementById(`save-btn-${commentId}`).addEventListener('click', () => {
    saveEditComment(commentId, originalContent);
  });
  document.getElementById(`cancel-btn-${commentId}`).addEventListener('click', () => {
    location.reload();
  });
};
```

> **쉽게 말하면:** 편지 봉투에 메모를 직접 쓰면(onclick) 특수문자가 봉투를 찢을 수 있어요. 대신 봉투 안에 별도 쪽지를 넣으면(addEventListener) 어떤 내용이든 안전해요!

프로젝트 전체에 **`window` 전역 함수가 10개**나 있어요!

**왜 개선하면 좋을까?**

`window`에 함수를 붙이는 것은 **전역 변수를 만드는 것과 같아요**. 이것이 왜 문제인지 구체적으로 설명할게요:

1. **이름 충돌 위험** — 다른 파일에서 같은 이름을 쓰면 나중에 정의된 것이 이전 것을 덮어씌워요. 예를 들어 `window.handleDeleteComment`를 다른 곳에서도 쓰면 둘 중 하나가 작동하지 않아요.

2. **추적이 어려움** — HTML에 `onclick="handleDeleteComment(5)"` 이렇게 써있을 때, 이 함수가 어디에 정의되어 있는지 찾기가 어려워요. `import`를 사용하면 에디터가 "정의로 이동"을 해줘요.

3. **결합도가 높아짐** — HTML(UI)이 JavaScript 함수 이름에 직접 의존해요. 함수 이름을 바꾸면 HTML도 다 찾아서 바꿔야 해요.

**추천:**

**이벤트 위임(Event Delegation)** 패턴을 사용하세요. `header.js`에서 이미 이 패턴을 쓰고 있어요!

```js

// boardDetailUI.js에서:

         class="text-[11px] text-gray-400 hover:text-[#05B29F]">수정</button>`
`<button data-action="delete-comment" data-comment-id="${comment.commentId}"
         class="text-[11px] text-gray-400 hover:text-red-500">삭제</button>`

// boardDetail.js — 이벤트 위임으로 한 곳에서 처리
function setupCommentEvents(postId) {

  if (!commentArea) return;

    if (action === 'edit-comment') toggleEditMode(commentId);
    if (action === 'delete-comment') await handleDeleteComment(commentId);
  });
}

// toggleEditMode, handleDeleteComment 등은 일반 함수로 (window. 빼기!)
function toggleEditMode(commentId) { ... }
async function handleDeleteComment(commentId) { ... }
```

> **응집도와 결합도 핵심 포인트:**
> - `window` 전역 = 높은 결합도 (HTML이 JS 함수 이름에 직접 의존)
> - 이벤트 위임 = 낮은 결합도 (HTML은 data 속성만, JS가 스스로 처리)

- 헤더 스타일을 바꾸려면 CSS 파일을 열어야 하는데, 다른 컴포넌트는 HTML의 클래스만 보면 돼요
- Tailwind의 반응형(`max-[1025px]:`)과 CSS의 `@media`가 섞이면 **우선순위 충돌**이 발생할 수 있어요
- CSS 파일은 "전역"이라 다른 페이지에도 영향을 줄 수 있어요

**추천:**

한꺼번에 바꾸기 어려우면, **새로 만드는 컴포넌트부터** Tailwind만 사용하고, 기존 CSS는 점진적으로 줄여가세요. 아예 없앨 수 없는 CSS(예: `@font-face`, 글로벌 리셋)만 남기는 것을 목표로 하면 돼요.

### 3-11. my.js — 회원탈퇴가 서버에 실제로 반영되지 않아요 (중요도: 중간)

**현재 상태:**

```js
// src/components/mypage/my.js 42~47번째 줄
btnWithdraw.addEventListener('click', () => {
  if (confirm('정말 탈퇴하시겠습니까? 모든 데이터가 삭제됩니다.')) {
    alert('회원탈퇴가 완료되었습니다.');
    window.location.href = '/';
    // ← API 호출이 없어요!
  }
});
```

**왜 개선하면 좋을까?**

사용자가 "탈퇴" 버튼을 누르면 "완료되었습니다"라고 나오지만, 실제로 서버에는 아무것도 전달되지 않아요. 사용자는 탈퇴했다고 생각하지만 **계정이 그대로 남아있어요**.

**추천:**

```js
btnWithdraw.addEventListener('click', async () => {
  if (!confirm('정말 탈퇴하시겠습니까? 모든 데이터가 삭제됩니다.')) return;
  
  try {
    const token = localStorage.getItem('token');
    const res = await fetch(`${BASE_URL}/user/me`, {
      method: 'DELETE',
      headers: { Authorization: `Bearer ${token}` },
    });
    if (!res.ok) throw new Error('회원탈퇴 실패');
    
    localStorage.removeItem('token');
    localStorage.removeItem('userId');
    alert('회원탈퇴가 완료되었습니다.');
    window.location.href = '/';
  } catch (error) {
    console.error(error);
    alert('회원탈퇴에 실패했습니다. 다시 시도해주세요.');
  }
});
```

> **쉽게 말하면:** 호텔 체크아웃할 때 프론트에 말만 하고 나가면, 호텔 시스템에는 아직 숙박 중인 것으로 남아요. 반드시 시스템(서버)에 "체크아웃합니다"를 전달해야 해요.

### 3-12. 인라인 style 속성 과다 사용 (중요도: 낮음)

**현재 상태:**

여러 파일에서 Tailwind 클래스와 인라인 `style` 속성을 혼용하고 있어요:

```js
// board.js 169~170번째 줄
postElement.className = 'bg-white rounded-2xl border border-[#00bba7]/15 shadow-sm ...';
postElement.style = 'display:flex; min-height:181px; padding:24px 24px 16px 24px; ...';
//                   ↑ 이 부분은 Tailwind으로 가능해요!
```

```html
<!-- mypage.html 22번째 줄 -->
<button ... style="display: flex; width: 100%; height: 57px; padding: 0 16px; ...">
<!-- Tailwind: class="flex w-full h-[57px] px-4 ..." -->
```

**왜 개선하면 좋을까?**

인라인 `style`은 **가장 높은 CSS 우선순위**를 가져서, 나중에 Tailwind 클래스로 스타일을 바꾸려 해도 `style`이 이겨요. 디버깅할 때도 "왜 내 Tailwind 클래스가 안 먹히지?" 하고 한참 찾다가 `style` 속성을 발견하게 돼요.

**추천:**

인라인 `style`을 Tailwind 클래스로 하나씩 옮겨보세요:

```js
// 수정 전
postElement.style = 'display:flex; min-height:181px; padding:24px; flex-direction:column;';

// 수정 후
postElement.className = '... flex min-h-[181px] p-6 flex-col';
```
