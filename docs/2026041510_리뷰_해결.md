## 2. `write.js` 카테고리 매핑 바로잡기

문제:

- `categoryMap`은 API 값 -> 한글
- 그런데 글쓰기에서는 한글 -> API 값으로 사용 중

즉 방향이 반대입니다.

### 어떻게 고치나요?

`write.js`에서는 `categoryMap`이 아니라 `reverseCategoryMap`을 써야 합니다.

### Before

```javascript
import { categoryMap } from '../../utils/boardConstants';

const apiCategory = categoryMap[selectedCategory] ?? selectedCategory;
```

### After

```javascript
import { categoryMap, reverseCategoryMap } from '../../utils/boardConstants.js';

const apiCategory = reverseCategoryMap[selectedCategory] ?? selectedCategory;
```

이 수정이 필요한 위치:

- 수정 버튼 처리
- 임시 저장 처리
- 작성 완료 처리

즉 `write.js` 안에서 `selectedCategory`를 API로 바꾸는 모든 곳입니다.

### 왜 이렇게 해야 하나요?

예를 들어 사용자가 `"공략"`을 선택하면:

- 화면에서는 `"공략"`이 맞고
- 서버에는 `"battle"`을 보내야 할 수 있습니다

그래서 이런 구조가 가장 안전합니다.

```javascript
// 서버 -> 화면
categoryMap[apiValue]

// 화면 -> 서버
reverseCategoryMap[label]
```

한쪽 맵을 억지로 양방향으로 쓰면 거의 반드시 한 번은 꼬입니다.

## 3. `dummyData` 의존 제거하기

문제:

- `board.js`는 더 이상 없는 `dummyData`를 fallback으로 사용 중

### 가장 쉬운 해결 방법

임시 데이터를 되살리는 것보다, **실패 시 빈 배열을 반환하고 빈 상태 UI를 보여주는 방식**이 더 좋습니다.

### Before

```javascript
const posts = data.data?.content ?? data.data ?? dummyData;
return Array.isArray(posts) ? posts : dummyData;
...
return dummyData;
```

### After

```javascript
const posts = data.data?.content ?? data.data ?? [];
return Array.isArray(posts) ? posts : [];
...
return [];
```

그리고 `renderPosts`에서 데이터가 없을 때 안내 문구를 보여주세요.

```javascript
if (sorted.length === 0) {
  postlist.innerHTML = `
    <div class="rounded-2xl bg-white p-10 text-center text-gray-400 shadow">
      아직 게시글이 없어요.
    </div>
  `;
  renderPagination([], 1);
  return;
}
```

### 왜 이 방법이 좋을까요?

- 없는 상수에 의존하지 않음
- API 실패와 빈 데이터 상황을 같은 흐름으로 다룰 수 있음
- 초보자도 흐름을 이해하기 쉬움

## 4. 댓글 수정 모드에서 `innerHTML`로 사용자 입력 넣지 않기

문제:

- 댓글 내용을 `textarea` HTML 문자열 안에 그대로 넣고 있음

이건 XSS 위험이 있습니다.

### 제일 안전한 방법

`innerHTML`로 textarea 전체를 만들지 말고, 실제 DOM 요소를 만들어 `value`에 넣으세요.

### Before

```javascript
contentP.innerHTML = `
  <textarea id="edit-input-${commentId}" rows="3">${originalContent}</textarea>
`;
```

### After

```javascript
contentP.replaceChildren();

const textarea = document.createElement('textarea');
textarea.id = `edit-input-${commentId}`;
textarea.rows = 3;
textarea.className =
  'w-full p-3 mt-2 border border-gray-200 rounded-lg focus:outline-none focus:border-[#05B29F] text-[14px] resize-none';
textarea.value = originalContent;

contentP.appendChild(textarea);
```

### 왜 `value`가 좋나요?

- 브라우저가 텍스트로 안전하게 넣어줍니다
- `</textarea>` 같은 문자열이 있어도 태그로 해석되지 않습니다
- 초보자도 의도를 이해하기 쉽습니다

## 5. `작성 완료`는 발행까지 끝난 뒤 이동해야 해요

문제:

- 지금은 `publishPost(postId)`를 호출만 하고 기다리지 않음

### Before

```javascript
if (postId) {
  publishPost(postId);
}
```

### After

```javascript
if (postId) {
  await publishPost(postId);
}
```

그리고 `publishPost()` 안에서 에러를 삼키지 말고 호출한 쪽으로 던져주는 것이 좋습니다.

### 더 좋은 형태

```javascript
export async function publishPost(postId) {
  const publishRes = await fetch(`${BASE_URL}/posts/${postId}/publish`, {
    method: 'PUT',
    headers: { Authorization: `Bearer ${getToken()}` },
  });

  if (!publishRes.ok) {
    throw new Error('게시글 발행 실패');
  }
}
```

이렇게 하면 `write.js`에서 실패 시 모달을 띄울 수 있습니다.

```javascript
try {
  await submitPost(...);
  history.pushState(null, '', '/board');
  window.loadPage();
} catch (error) {
  await showModal('오류', error.message || '게시글 작성에 실패했습니다.', 'danger');
}
```

### 핵심 원칙

버튼 이름이 `작성 완료`면, 정말 완료된 뒤에만 다음 화면으로 넘어가야 합니다.

## 6. 상세 페이지 이벤트는 `document` 말고 상세 영역에 묶으세요

문제:

- `boardDetail.js`가 `document.addEventListener('click', ...)`를 매번 추가함

### 해결 방법

상세 페이지 루트 컨테이너 하나를 잡고, 그 안에서만 이벤트 위임하세요.

### Before

```javascript
document.addEventListener('click', async (e) => {
  const action = e.target.dataset.action;
  ...
});
```

### After

```javascript
function setupDetailActions() {
  const contentArea = document.getElementById('postDetailContent');
  const commentArea = document.getElementById('commentSection');
  const root = contentArea?.parentElement;
  if (!root) return;

  root.onclick = async (e) => {
    const target = e.target.closest('[data-action]');
    if (!target) return;

    const action = target.dataset.action;

    if (action === 'edit-comment') {
      toggleEditMode(target.dataset.commentId);
      return;
    }

    if (action === 'delete-comment') {
      await handleDeleteComment(target.dataset.commentId);
      return;
    }

    if (action === 'edit-post') {
      handleEditPost(target.dataset.postId);
      return;
    }

    if (action === 'delete-post') {
      await handleDeletePost(target.dataset.postId);
    }
  };
}
```

### 왜 좋아지나요?

- 상세 페이지 안에서만 동작
- 페이지를 다시 열어도 리스너가 전역으로 쌓이지 않음
- 결합도가 낮아짐

## 7. 파티 프리셋의 성별 값은 서버 값을 그대로 쓰세요

문제:

- 저장된 프리셋을 불러올 때 `gender: "man"`으로 고정

### Before

```javascript
presets = list.map((p) => ({
  apiId: p.partyId,
  name: p.deckname,
  gender: "man",
  pokemonIds: p.pocketmons,
  ...
}));
```

### After

```javascript
presets = list.map((p) => ({
  apiId: p.partyId,
  name: p.deckname,
  gender: p.gender ?? 'man',
  pokemonIds: p.pocketmons,
  party: p.pocketmons.map((id) =>
    pokemons.find((pk) => pk.id === Number(id)) || null,
  ),
}));
```

### 왜 `?? 'man'`을 쓰나요?

- 서버가 정말 값을 안 줄 때만 기본값 사용
- 정상 데이터는 그대로 유지

이 패턴은 초보자에게 아주 중요합니다.

- 기본값은 "없을 때만"
- 있는 값을 덮어쓰면 안 됩니다

## 8. API 파일은 "데이터만", 화면 파일은 "화면만" 다루기

지금 `api/post.js`는 API 모듈인데 UI까지 만지고 있습니다.

이건 꼭 정리하는 게 좋습니다.

### 나쁜 예

```javascript
document.getElementById("commentInput").value = "";
location.reload();
history.pushState(null, "", `/board/${postId}`);
window.loadPage();
```

### 좋은 구조

`api/post.js`

```javascript
export async function postComment(postId, text) {
  const res = await fetch(...);
  if (!res.ok) throw new Error('댓글 작성 실패');
  return res.json();
}
```

`boardDetail.js`

```javascript
await postComment(postId, text);
document.getElementById('commentInput').value = '';
await initPostDetail(postId);
```

### 왜 이렇게 나누나요?

- API 파일은 재사용하기 쉬움
- 테스트하기 쉬움
- 어떤 화면에서든 원하는 방식으로 결과를 처리할 수 있음

이게 바로 **응집도를 높이고 결합도를 낮추는 기본 패턴**입니다.

## 9. Tailwind로 정리할 때 기준

이번 프로젝트는 CSS를 최소화하고 Tailwind 위주로 가는 것이 목표입니다.

그러면 아래 기준을 추천합니다.

### 1. `style="..."`보다 Tailwind 클래스 우선

예:

```html
class="rounded-xl border border-gray-200 px-4 py-3 text-sm text-gray-500"
```

### 2. `onmouseover`, `onmouseout` 대신 hover 클래스 사용

### Before

```html
<button
  onmouseover="this.style.borderColor='#5eead4'"
  onmouseout="this.style.borderColor='#d1d5db'">
</button>
```

### After

```html
<button
  class="border border-gray-300 bg-slate-50 text-gray-400 hover:border-teal-300 hover:bg-teal-50 hover:text-teal-600 transition-colors">
</button>
```

### 3. 정말 반복되는 스타일만 최소 CSS로

예를 들어 포켓몬 모달처럼 구조가 너무 복잡해서 Tailwind만으로 읽기 어려우면,

- 공통 클래스 1~2개만 CSS로 뽑고
- 나머지는 Tailwind로 유지

이 정도가 좋습니다.

## 12. 마무리

이번 변경은 리팩터링 방향 자체는 좋았습니다.  
다만 지금 단계에서는 "나누는 것"보다 더 중요한 "역할을 정확히 나누는 것"이 아직 부족합니다.

꼭 기억하면 좋은 한 줄 정리:

- 상수는 방향이 중요하다
- API는 데이터를 반환하고 끝내야 한다
- 사용자 입력은 `innerHTML`로 다시 넣지 않는다
- SPA에서는 전역 이벤트를 쉽게 달지 않는다
- Tailwind를 쓰기로 했으면 JS 스타일 조작은 최소화한다

이 기준으로 한 번 더 정리하면 다음 리뷰에서는 훨씬 안정적인 코드가 나올 수 있습니다.
