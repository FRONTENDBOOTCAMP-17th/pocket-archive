## 0. 이번 리뷰 범위

이번 리뷰는 `develop` 브랜치에서 `1471e19` 이후 `ed2d2db`까지 들어온 변경사항을 기준으로 했습니다.

이번에 특히 많이 바뀐 곳은 아래입니다.

- `src/api/post.js`, `src/api/pokemonApi.js`, `src/api/user.js` — API 로직 분리
- `src/components/board/*` — 게시판 목록, 상세, 글쓰기
- `src/components/myparty/*` — 나의 파티 만들기 리팩터링
- `src/components/pokedex/*` — 도감 이벤트 위임, 북마크 처리
- `src/components/mypage/*` — 마이페이지와 포켓몬 모달 정리
- `src/utils/boardConstants.js` — 게시판 카테고리 상수 분리

확인한 내용:

- `npm install` 후 `npm run build`는 성공했습니다.
- 다만 빌드는 "문법"과 "번들링" 위주 검사라서, 이번 리뷰에서는 **런타임 버그**, **응집도/결합도 문제**, **Tailwind 사용 원칙 위반**을 더 중요하게 봤습니다.

## 2. 먼저 칭찬할 점

이번 변경에는 분명 좋은 방향도 많았습니다.

- API 코드를 `src/api/`로 모으려는 시도는 좋았습니다.
- `window.selectPokemon` 같은 전역 함수 일부를 이벤트 위임으로 바꾼 것은 좋은 방향입니다.
- `showModal`, `showConfirm`, `showPrompt`처럼 공통 모달을 만들기 시작한 점도 좋습니다.
- `boardConstants.js`로 게시판 상수를 분리한 것 역시 "중복 제거" 관점에서는 맞는 시도입니다.

문제는 "분리하려는 방향"은 좋았는데, 아직 중간 단계라서 **역할이 완전히 정리되지 않았고**, 그 과정에서 몇 개의 회귀(regression)가 생겼다는 점입니다.

## 3. 자세한 리뷰

### 🔴 1-1. `write.js`의 카테고리 매핑 방향이 뒤집혔어요

파일:

- `src/components/board/write.js:171`
- `src/components/board/write.js:195`
- `src/components/board/write.js:217`
- `src/utils/boardConstants.js:2-14`

현재 코드:

```javascript
const apiCategory = categoryMap[selectedCategory] ?? selectedCategory;
```

그런데 `boardConstants.js`의 `categoryMap`은 지금 이렇게 정의되어 있습니다.

```javascript
export const categoryMap = {
  free: '자유게시판',
  guide: '질문게시판',
  battle: '공략',
  party: '파티공유',
};
```

즉 이 맵은 **API 값 -> 화면용 한글** 방향입니다.  
그런데 `write.js`에서는 사용자가 고른 한글 값(`자유게시판`, `공략` 등)을 API 값으로 바꾸려고 같은 맵을 쓰고 있어요.

이 경우 실제로는:

- `selectedCategory === "자유게시판"`
- `categoryMap["자유게시판"] === undefined`
- 그래서 결국 `"자유게시판"` 그대로 서버에 전달

이전 버전에는 이 용도로 `reverseCategoryMap`이 있었는데, 이번 리팩터링에서 그 부분이 잘못 바뀌었습니다.

왜 중요할까요?

- 글 작성
- 임시 저장
- 글 수정

세 경로가 모두 같은 실수를 하고 있어서, 게시판 카테고리 저장이 틀어질 수 있습니다.

이건 초보 개발자가 많이 하는 실수예요.  
"표시용 값"과 "서버용 값"은 꼭 분리해야 합니다.

### 🔴 1-2. `board.js`의 fallback이 오히려 게시판을 터뜨려요

파일:

- `src/components/board/board.js:10-15`

현재 코드:

```javascript
const posts = data.data?.content ?? data.data ?? dummyData;
return Array.isArray(posts) ? posts : dummyData;
...
return dummyData;
```

문제는 `dummyData`가 이 파일 안에 더 이상 없습니다.

즉 API가 실패하거나, 응답 구조가 예상과 다르면 "임시 데이터로 보여주기"가 아니라 바로 `ReferenceError: dummyData is not defined`가 납니다.

이건 리팩터링에서 자주 생기는 전형적인 버그입니다.

- 예전에는 있던 상수를 지웠다
- 그런데 fallback 코드에서 이름은 그대로 남았다
- 평소에는 API가 잘 돼서 못 찾는다
- 배포 후 장애 상황에서만 터진다

즉, **정상 상황에서는 안 보이고 장애 상황에서만 보이는 더 위험한 버그**입니다.

### 🔴 1-3. 댓글 수정 모드에서 XSS 가능성이 다시 생겼어요

파일:

- `src/components/board/boardDetail.js:176-183`

현재 코드:

```javascript
const originalContent = contentP.innerText;

contentP.innerHTML = `
  <textarea id="edit-input-${commentId}"
    ...
    rows="3">${originalContent}</textarea>
`;
```

겉으로 보면 `<textarea>` 안에 넣으니 안전해 보이지만, 그렇지 않습니다.

예를 들어 누군가 댓글에 이런 문자열을 썼다고 가정해볼게요.

```html
</textarea><img src=x onerror=alert(1)>
```

그러면 수정 모드로 바꾸는 순간 `<textarea>`가 닫히고 뒤의 HTML이 실제 DOM으로 들어갈 수 있습니다.

핵심은 이것입니다.

- 화면 출력할 때 `escapeHtml`을 했더라도
- `innerText`로 다시 꺼내면 원래 텍스트가 복원되고
- 그걸 다시 `innerHTML`에 넣으면 다시 위험해집니다

즉, **사용자 입력을 `innerHTML` 문자열 안으로 다시 밀어 넣으면 안 됩니다.**

이 부분은 이전 리뷰에서도 비슷한 위험을 지적했는데, 이번 버전에서는 인라인 `onclick`은 줄었지만 같은 문제가 다른 형태로 남아 있습니다.

### 🟠 1-4. `작성 완료`가 실제 발행 완료를 기다리지 않아요

파일:

- `src/components/board/write.js:15-17`
- `src/components/board/write.js:216-220`
- `src/api/post.js:169-178`

현재 코드:

```javascript
if (postId) {
  publishPost(postId);
}
```

여기서 `publishPost`는 `async` 함수인데 `await`가 없습니다.

그러면 실제 흐름은 이렇게 됩니다.

1. 글 초안 생성
2. 발행 요청 시작
3. 기다리지 않고 바로 `/board`로 이동

즉 사용자는 "작성 완료"를 눌렀는데:

- 발행이 아직 안 끝났을 수도 있고
- 발행이 실패했는데도 화면은 그냥 넘어갈 수 있습니다

특히 `publishPost()` 안에서도 에러를 내부에서 `console.error`만 하고 끝내기 때문에, 호출한 쪽에서는 실패를 감지하기 어렵습니다.

결과적으로 "작성 완료" 버튼 이름과 실제 동작이 어긋납니다.

### 🟠 2-1. `boardDetail.js`가 페이지 진입마다 전역 클릭 리스너를 계속 등록해요

파일:

- `src/components/board/boardDetail.js:94-135`
- `src/main.js:93-95`

현재 코드:

```javascript
document.addEventListener('click', async (e) => {
  const action = e.target.dataset.action;
  ...
});
```

`initPostDetail()`은 라우팅 때마다 다시 호출됩니다.  
그런데 이 전역 리스너는 제거되지 않습니다.

즉 사용자가:

1. 게시글 상세에 들어간다
2. 다른 페이지로 갔다가
3. 또 다른 게시글 상세에 들어간다

이 과정을 반복하면 `document`에 같은 리스너가 계속 쌓입니다.

그러면 나중에는:

- 삭제 버튼 한 번 눌렀는데 로직이 여러 번 실행되거나
- 수정 모드가 중복 실행되거나
- 디버깅하기 어려운 "가끔 두 번 실행됨" 문제가 생깁니다

이게 바로 결합도가 높은 코드의 특징입니다.

- 컴포넌트 하나가 자기 영역만 듣지 않고
- 문서 전체(`document`)를 잡아버리면
- 다른 페이지 상태와 섞이기 시작합니다

SPA에서는 특히 더 조심해야 합니다.

### 🟠 2-2. 저장된 파티 프리셋의 성별 정보를 버리고 있어요

파일:

- `src/components/myparty/myparty.js:89-95`

현재 코드:

```javascript
presets = list.map((p) => ({
  apiId: p.partyId,
  name: p.deckname,
  gender: "man",
  pokemonIds: p.pocketmons,
```

여기서는 서버에서 받은 프리셋을 다시 로컬 상태로 바꾸면서 `gender`를 무조건 `"man"`으로 넣고 있습니다.

그러면 사용자가 여자 트레이너로 저장한 프리셋도:

- 새로고침 후 다시 불러오면
- 항상 남자 트레이너 카드로 보일 가능성이 큽니다

이런 문제는 눈에 띄는 큰 에러는 아니어서 더 위험합니다.

- 저장은 되는 것처럼 보이고
- 일부 데이터만 조용히 바뀝니다

즉 "기능은 되는 것 같은데 왜 저장한 값이 다르게 보이지?"가 됩니다.

### 🟡 3-1. `api/post.js`가 API 모듈 역할을 벗어나 있어요

파일 예시:

- `src/api/post.js:41-42`
- `src/api/post.js:63`
- `src/api/post.js:82`
- `src/api/post.js:96`
- `src/api/post.js:205-210`
- `src/api/post.js:260-261`

예시 코드:

```javascript
document.getElementById("commentInput").value = "";
location.reload();
...
location.href = "/board";
...
history.pushState(null, "", `/board/${postId}`);
window.loadPage();
```

`src/api/post.js`라는 이름을 보면 보통 기대하는 역할은 이것입니다.

- 서버에 요청 보내기
- 응답 데이터 반환하기
- 실패하면 에러 던지기

그런데 지금은 여기에:

- DOM 접근
- 새로고침
- 페이지 이동
- 라우터 호출

까지 다 들어 있습니다.

이러면 어떤 문제가 생길까요?

- API 함수를 다른 화면에서 재사용하기 어렵습니다
- 테스트하기 어렵습니다
- 같은 함수를 불렀는데 화면까지 강제로 바뀝니다
- HTML id가 바뀌면 API 파일도 같이 깨집니다

즉 응집도는 낮아지고, 결합도는 높아집니다.

### 🟡 3-2. Tailwind를 쓰기로 했는데 인라인 스타일과 인라인 이벤트가 아직 많아요

파일 예시:

- `src/components/myparty/mypartyUI.js:67-75`

- `pages/pokedex.html:52`

예시:

```html
onmouseover="this.style.borderColor='#5eead4'; ..."
onmouseout="this.style.borderColor='#d1d5db'; ..."
```

그리고 `style="..."`도 여전히 매우 많이 남아 있습니다.

이번 과제 조건은 분명합니다.

- CSS는 최소한만 사용
- 가능하면 Tailwind로 작성

그런데 지금 구조는:

- Tailwind 클래스
- 인라인 `style`
- 인라인 이벤트

이 섞여 있습니다.

이렇게 되면 초보 개발자가 가장 힘들어집니다.

- "색이 어디서 바뀌는지" 찾기 어렵고
- hover 상태가 CSS인지 JS인지 헷갈리고
- 같은 UI를 재사용하기가 어려워집니다

Tailwind를 쓰는 이유는 "스타일을 한눈에 보이게" 하려는 것인데, 지금은 그 장점이 많이 흐려져 있습니다.

## 4. 이번 리뷰의 핵심 메시지

이번 변경의 가장 큰 주제는 이 두 가지입니다.

### 1. 리팩터링은 "파일만 옮기는 것"이 아니에요

API 분리, 상수 분리, UI 분리는 좋은 방향입니다.  
하지만 파일을 나눴다고 자동으로 응집도가 높아지지는 않습니다.

정말 중요한 건:

- 이 파일이 **한 가지 역할만 하는지**
- 다른 파일의 DOM 구조, 라우터, 전역 상태를 **얼마나 적게 아는지**

입니다.

### 2. "화면용 값"과 "서버용 값"을 꼭 구분해야 해요

이번 리뷰에서 가장 치명적인 버그가 카테고리 매핑에서 나왔습니다.

- 사용자에게 보여주는 값
- 서버에 보내는 값

이 두 개는 다를 수 있습니다.  
이걸 섞으면 지금처럼 글 작성/수정 기능 전체가 조용히 틀어질 수 있어요.

## 6. 마무리

이번 코드는 "좋은 방향으로 리팩터링하려는 시도"가 분명히 보였습니다.  
다만 아직은 **책임 분리의 기준이 완전히 잡히지 않아서**, 나눈 파일들 사이에 역할이 다시 섞이고 있습니다.

특히 초보 단계에서는 아래 원칙을 꼭 기억하면 좋습니다.

- API 파일은 데이터만 다룬다
- UI 파일은 화면만 그린다
- 이벤트 파일은 클릭/입력만 다룬다
- 화면용 문자열과 서버용 값은 분리한다
- 사용자 입력은 절대 `innerHTML` 문자열에 그대로 넣지 않는다

다음 문서인 [2026041510_리뷰_해결.md](./2026041510_리뷰_해결.md)에는 위 문제들을 어떻게 고치면 되는지 단계별로 정리해두겠습니다.
